# 12 - 上下文压缩与错误恢复

本章你将学会：应对长对话爆上下文、应对 API 报错——这些框架已内置，你只需了解并按需调参。

## 1. 上下文压缩（Compaction）

Agent 跑久了，对话历史会越来越长，超过模型的上下文上限。MochiKit 在每轮思考前自动跑一条
**分层压缩流水线**（从便宜到昂贵）：

1. `ToolResultBudget`：把过长的工具结果截断（默认 4000 字符）。
2. `MicroCompaction`：只保留最近 3 条工具结果，更早的替换为占位提示。
3. `SnipCompaction`：对话太长时，保留首条 + 最近 6 条，中间折叠成一句提示。

这些都是**零 API 调用**的本地操作。如果还不够，框架会触发反应式压缩（见下）。

你通常无需操心。如需自定义：

```ts
import { Agent, CompactionPipeline, ToolResultBudget, MicroCompaction, SnipCompaction } from 'mochikit';

const myPipeline = new CompactionPipeline([
  new ToolResultBudget(8000),   // 工具结果最多 8000 字符
  new MicroCompaction(5),       // 保留最近 5 条工具结果
  new SnipCompaction(10),       // 保留首条 + 最近 10 条
]);

const agent = new Agent({ /* ... */ compaction: myPipeline });
```

## 2. 错误恢复（Recovery）

调用 LLM 可能遇到三类错误，框架的 `Recovery` 自动处理：

| 错误 | 处理 |
|---|---|
| 429 / 529（限流/过载） | 指数退避重试（带 jitter），连续过载切换到 `fallbackModel` |
| prompt too long（上下文超长） | 反应式压缩一次（只留最近 5 条），再重试 |
| max_tokens（输出被截断） | 自动升级 max_tokens 重试，必要时追加“继续”提示 |

默认重试 5 次，退避基数 500ms、上限 32s。

配置：

```ts
import { Agent, Recovery } from 'mochikit';

const recovery = new Recovery({
  maxRetries: 5,
  baseDelayMs: 500,
  maxDelayMs: 32_000,
  fallbackModel: 'glm-4.7-mini', // 可选：过载时降级到的模型
});

const agent = new Agent({
  // ...
  recovery,
  fallbackModel: 'glm-4.7-mini',
});
```

## 3. 这些是自动的

`Agent` 默认就装了默认压缩流水线和 Recovery。**大多数场景你什么都不用配**，
只有在需要调参（更长历史、不同退避、备用模型）时才显式传入。

## 4. 手动压缩（高级）

如果你在外部管理消息历史，可以直接用压缩函数：

```ts
import { reactiveCompact, defaultPipeline } from 'mochikit';

const pipeline = defaultPipeline();
const compacted = pipeline.compact(messages);   // 跑完整流水线
const emergency = reactiveCompact(messages, 5); // 紧急只留 5 条
```
